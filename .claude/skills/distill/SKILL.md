# /distill —— 萃取参考项目

把一个参考 agent 项目的**思想**（不是代码）增量并入本工程。
方法论出处：`docs/progress/principles.md` 的"萃取四步"，执行前先读它和
`docs/progress/module-map.md`（六层定义）。

## 前置检查

1. **命名红线**：给参考项目起一个中性代号（如 `agent-server`），后续所有产出
   （文档、注释、提交信息）只用代号。真实名称、本地路径、禁用词写进
   `.claude/local-notes.md`（已 gitignore），**绝不写入任何会提交的文件**。
2. 确认能访问参考项目源码（路径记在 local-notes.md）。

## 四步流程

### 第 1 步：分层

快速浏览参考项目的目录结构（不深入读代码），按本工程六层模型把它拆成两堆：

- **本质堆（要）**：对应 B 模型层 / C 工具层 / D Agent 核心——agent 循环、
  工具设计、上下文/记忆、鲁棒性、提示词。
- **附带堆（砍）**：对应 E 会话调度 / F 服务端——多租户、WebSocket、worker、
  鉴权、限流等产品脚手架。**默认跳过，别被体量吓到。**

产出：一张"该项目 → 六层"的映射表，注明每层在该项目里的源码位置
（位置速查写入 local-notes.md）。

### 第 2 步：只读本质堆

只深入读本质堆，重点回答四个问题：

- 它的 agent 主循环怎么组织？（思考→工具→观察的具体做法）
- 它的工具怎么定义、注册、执行、隔离？
- 它怎么管理上下文/记忆/裁剪？
- 它怎么处理失败？（循环检测、重试、修复）

### 第 3 步：推导，不照抄

对每个值得吸收的思想，回答"它为什么这么设计、换成 Go 的最干净写法是什么"。
真正的产物是认知，代码只是副产品。产出一份萃取计划：

- 哪些思想 → 并入本工程哪个模块（对照 `docs/progress/status.md` 的表）
- 哪些本工程已有更好做法，不吸收（写明理由）
- 实施顺序建议

### 第 4 步：增量并入

按计划实施。每并入一块，在 `status.md` 对应行的"说明"里记下
"此思想来自 <代号> 的 <模块>"。全部完成后执行 `/sync-progress` 收尾。

## 提醒义务

过程中若发现用户想吸收 E/F 类（附带复杂度）内容，先提醒
"这偏离了'只做 agent 本质'的纲领"，确认后再做。
