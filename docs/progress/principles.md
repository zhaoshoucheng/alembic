# 纲领：只萃取 agent 的本质复杂度

> 这是本工程的"北极星"。每次会话、每次决定做什么，都以本文件为准绳。
> 如果某个工作偏离了这里写的方向，**Claude 有义务主动提醒我（用户）"这偏离了本质"**。

## 北极星目标

**项目名：Alembic**（炼金术里的蒸馏器）—— *distill the essence of every agent*。
寓意：一个能**吸收、提炼其它各种 agent、取长补短**的 agent；它从别人身上萃取的不是
代码，而是本质。

做一个**本地、单用户、单个可执行文件**的终端 agent——形态对标 Claude Code：
编译成一个二进制（`alembic`），直接在终端里交互（输入→流式回答→再输入），不依赖服务器。

## 唯一关心的东西：agent 的本质

我学习的目的是**理解并实现 agent 的本质复杂度**，不是复刻一个产品。
区分两种复杂度，是本工程一切取舍的总开关：

- **本质复杂度（essential，要）** — agent 之所以是 agent 的东西：
  怎么思考、怎么调用工具、怎么管理上下文/记忆、怎么从错误中恢复、怎么写提示词。
  对应模块层的 **B 模型层 / C 工具层 / D Agent 核心**。
- **附带复杂度（accidental，砍）** — 只为"做成一个能卖钱的多用户产品"才需要的脚手架：
  多租户、WebSocket 会话、worker 分发、鉴权、限流、可观测性、服务端 API。
  对应模块层的 **E 会话与调度 / F 服务端**——**本工程明确不做**。

> 一句话：盯住 **B/C/D**，尤其是 **D（agent 的大脑）和 C（工具）**；凡是闻到 E/F 味道的，
> 默认跳过，除非有明确理由。

## 方法：遇到任何 agent 项目，都这样"萃取"

这不是一次性的事。以后我还会遇到别的 agent 开源/参考项目，每个都用同一套手法，
把它们的本质**增量补充进本工程**：

1. **先分层**：把目标项目快速拆成"本质 vs 附带"两堆，别被它的体量吓到。
2. **只读本质那堆**：聚焦它的 agent 循环、工具设计、上下文管理、鲁棒性处理。
3. **再推导、用 Go 重写**：不照抄，要读懂后用最干净的方式重新实现——
   真正的产物是"长在我脑子里的认知"，代码只是副产品。
4. **增量并入**：把新学到的本质思想补充进当前工程对应的 B/C/D 模块，
   并在 `status.md` 记下"这部分思想来自哪个参考项目的什么模块"。

> 参考项目对我的价值永远是**思想**，不是它的架构或代码量。它们替我把生产里会踩的坑
> 都踩过了，我只取其中"关于智能体本身"的那部分。

## Claude 的提醒义务（写给每次会话的我）

- 当用户开始做、或要求做 **E/F 类（会话/服务端/多租户/分发/鉴权…）** 的工作时，
  先**提醒**："这属于附带复杂度，偏离了'只做 agent 本质'的纲领，确认要做吗？"
- 当一个任务能用更小、更聚焦本质的方式完成时，倾向于那个更小的方式。
- 引入新参考项目时，主动按上面的"萃取四步"帮用户分层，先圈出本质再动手。
